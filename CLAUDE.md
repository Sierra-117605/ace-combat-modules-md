# CLAUDE.md

本ドキュメントは Claude Code 向けの指示書である。
Claude Code はリポジトリでの作業開始時に必ず本書を読み込み、
本書のルールに従って動作する。

---

## プロジェクト概要

このリポジトリは HOI4 mod「Millennium Dawn」向けの
ACモジュールMOD (ACM-MD) の開発リポジトリである。

詳細は `MOD_OVERVIEW.md` を参照すること。

---

## 開発者の特性

- ネットワークエンジニア(非プログラマ)
- 主要言語: Python、Java(ただしコードを書く能力は無く、AIによる支援が前提)
- コードレビューは「挙動」ベースで行い、コード自体は読まない
- ハルシネーション混入を強く嫌う

---

## 必読ドキュメント

作業開始時に以下を必ず読み込むこと:

1. `MOD_OVERVIEW.md` — プロジェクト全体の地図
2. `PLAN.md` — やりたいこと
3. `SPEC.md` — 仕様
4. `TODO.md` — 現在のタスク
5. `KNOWLEDGE.md` — 既存の知見・地雷
6. `DATABASE.md`(存在する場合) — 兵器の根拠台帳
7. `INSTRUCTION_FOR_DATABASE.md`(DB作成タスクの場合) — DB作成指示書

## 厳守ルール

### 1. ハルシネーション禁止

- 推測で実装・記述しない
- 「~らしい」「~と思われる」「~の可能性が高い」等の表現を避ける
- 不確実な点は必ず確認した上で記述する
- 確認できない情報は「未確認」と明示する

### 2. 典拠ベースの作業

- AC作品の設定情報は必ず一次情報源を当たる
- **超兵器分類は公式(acecombat.jp の ACES CHRONICLE)を最優先**
  - acecombat.wiki.gg の Disputed セクションの独自判定は採用しない
- それ以外の兵器情報は acecombat.wiki.gg を優先
- 個人ブログ、Steam Workshop投稿者コメント、Reddit、ファン創作Wiki等は
  典拠にしない
- 出典URLは可能な限り残す

### 3. MDのコード調査前に実装しない

- MD はバニラから大幅独自実装のため、推測実装は禁止
- 必ず以下の二段構えで作業する:
  - Phase 1: 実コード調査(grepでファイルパス+行番号を残す)
  - Phase 2: 調査結果を踏まえて実装
- 調査結果は KNOWLEDGE.md に蓄積する

### 4. ドキュメント更新義務

- 仕様変更があれば SPEC.md を更新
- タスク完了/追加があれば TODO.md を更新
- 新しい知見・ハマりポイントを得たら KNOWLEDGE.md に追記
- 兵器情報を新規追加/更新したら DATABASE.md を更新
- コミット時にこれらが整合しているか確認

### 5. 日本語メイン

- ドキュメントは日本語をメインで記述
- 固有名詞は「日本語(英語原語)」併記
- 例: アリコーン (Alicorn)、ストーンヘンジ (Stonehenge)
- 典拠URL、実コードのファイル名・キーワードは英語のまま

### 6. 報告と承認

- タスク完了時には必ず進捗報告
- 判断が分かれそうな点では止まって質問する
- 勝手に判定しない
- 特に判断を仰ぐべきケース:
  - 設定上「量産されなかった1点物」かつ「機体本体」のもの
  - MD既存実装と機能が被るもの
  - 典拠が公式と二次創作で食い違うもの

### 7. 並行作業の禁止

- Claude Code と Codex の同時並行作業は禁止する
- 作業前に必ず `git pull` で最新を取得する
- 作業完了時に `git push` を行ってから手放す
- 別のエージェントに引き継ぐ場合は、コミット履歴で状況を確認する
- 大きな変更を入れる前に状況確認のためコミット履歴・ブランチ状態を読む

### 8. 役割分担とハンドオフ(instructions/ 方式)

本プロジェクトでは Claude Code が PM/PL(方針決定・タスク詳細化・レビュー)、
Codex が実装担当という分担を基本とする。情報共有は会話の引き継ぎではなく
**Git上のファイル**で行う。

#### Claude Code(方針決定側)の役割

- Codexに実装を依頼するタスクが発生したら、着手前に
  `instructions/YYYY-MM-DD_<タスク概要>.md` を作成する
- 指示書には以下を含める:
  - 目的(なぜこの作業をするか)
  - 具体的な作業内容(何を、どのファイルに、どんな構成で)
  - 典拠・参照すべきドキュメント(DATABASE.md, KNOWLEDGE.md の該当箇所等)
  - 完了条件(何ができていれば完了か)
  - 判断に迷ったら止まって質問すべきケース
- 指示書はCodexが**会話の文脈無しで読むだけで着手できる**ことを基準に書く
- Codexのcommit後、`git log` / `git diff` で結果を確認してからレビュー・
  次の指示書作成に進む

#### Codex(実装側)の役割

- 作業開始時、`instructions/` フォルダ内の自分宛ての未対応指示書を確認する
- 指示書に従って実装し、commitメッセージに対応する指示書ファイル名を
  含める(例: `[Codex] DATABASE.md 超兵器セクション作成 (ref: instructions/2026-06-27_database_md_superweapons.md)`)
- 指示書の内容だけで判断できない場合は、自己判断で進めず
  TODO.md の「ブロック中のタスク」に追記して止まる
- 完了した指示書は `instructions/done/` に移動する(履歴として残す)

#### instructions/ フォルダ構成

```
instructions/
├ done/                          完了済み指示書(移動先)
└ YYYY-MM-DD_<タスク概要>.md     未対応の指示書
```

## スコープ外(厳守)

以下は本MODでは扱わない:

- AC作品の通常兵装(4AAM、6AAM、QAAM、SAAM、マルチロックオン系)
- 固定設置型超兵器のモジュール化(ストーンヘンジ、エクスキャリバー等)
- 量産戦闘機の本体追加
- 機体・艦・車両本体のアイコン新規作成
- 架空車両・実在車両・実在艦のDB化

---

## コーディング規約

- HOI4 modの標準ディレクトリ構造に従う
- モジュールIDの接頭辞: `acm_`(ACM-MDの略)
- 既存MDモジュールと命名衝突しないことを `grep` で確認する
- ファイル名・ディレクトリ名はMDの命名規則に追従する
- ローカライゼーションは英語と日本語の両方を用意する

## 困ったとき

- 仕様の優先順位は: 厳守ルール > SPEC.md > PLAN.md
- 判断に迷ったら勝手に進めず、ユーザに質問する
- 致命的なミス(コード破壊、誤コミット等)を起こしそうなら必ず止まる
